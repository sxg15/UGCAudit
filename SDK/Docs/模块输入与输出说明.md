# 模块输入与输出说明

本文档说明模块运行时会收到什么，以及必须写出什么。

## 运行方式

客户端启动模块时，会把输入和输出文件路径传给模块：

```text
main.py --input input.json --output output.json
```

如果 `module.json` 的 `launch.args` 里没有写 `{inputJson}` 或 `{outputJson}`，客户端也会自动补上。但建议模块开发者明确写出来，方便本地调试。

## input.json 总体结构

模块运行前，客户端会写出 `input.json`。常见结构如下：

```json
{
  "runId": "run_20260601_001",
  "stepId": "text_safety_1",
  "moduleId": "custom.demo.text-risk",
  "moduleKind": "text_safety",
  "moduleName": "示例文本审核",
  "workDir": "D:/UGCAuditData/runs/run_20260601_001/steps/text_safety_1",
  "taskName": "每日审核任务",
  "artifactDir": "D:/UGCAuditArtifacts/每日审核任务/run_20260601_001",
  "stepArtifactDir": "D:/UGCAuditArtifacts/每日审核任务/run_20260601_001/steps/text_safety_1/artifacts",
  "resourceRoot": "D:/UGCAuditData/resources",
  "inputs": {},
  "files": [],
  "params": {},
  "modelPath": "",
  "previous": {}
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `runId` | 本次运行 ID。 |
| `stepId` | 当前流程步骤 ID。 |
| `moduleId` | 当前模块 ID。 |
| `moduleKind` | 当前模块类型。 |
| `moduleName` | 当前模块名称。 |
| `workDir` | 当前步骤工作目录。 |
| `taskName` | 本次任务名称。 |
| `artifactDir` | 本次任务的产物目录。 |
| `stepArtifactDir` | 当前步骤专用产物目录。 |
| `resourceRoot` | 本次运行资源根目录。 |
| `inputs` | 数据线传入的数据集合。 |
| `files` | 当前步骤可处理的文件列表。 |
| `params` | 用户在流程节点上填写的参数。 |
| `modelPath` | 模型目录。没有模型时为空字符串。 |
| `previous` | 上游步骤的原始输出结果。 |

实时运行时，`input.json` 还可能包含：

| 字段 | 说明 |
| --- | --- |
| `progressPath` | 模块可以追加进度 JSON 行。 |
| `cancelPath` | 用户中断任务时，客户端会创建这个文件。 |

## 图片集合输入

图片集合会放在 `inputs.images`、`inputs.images-a` 或 `inputs.images-b` 下，具体名字取决于连接到哪个端口。

```json
{
  "inputs": {
    "images": {
      "dataType": "imageCollection",
      "items": [
        {
          "path": "D:/Project/assets/a.png",
          "name": "a.png",
          "extension": "png",
          "sourceAssetId": "asset_1",
          "sourceAssetName": "待测项目",
          "relativePath": "assets/a.png"
        }
      ]
    }
  }
}
```

图片模块通常读取：

- `path`：图片文件路径。
- `name`：文件名。
- `relativePath`：相对待测项目的路径。

## 文本集合输入

文本集合会放在 `inputs.texts`、`inputs.texts-a` 或 `inputs.texts-b` 下。

```json
{
  "inputs": {
    "texts": {
      "dataType": "textCollection",
      "items": [
        {
          "sourceType": "file",
          "path": "D:/Project/config/dialogue.txt",
          "name": "dialogue.txt",
          "relativePath": "config/dialogue.txt",
          "text": "这里是文件内容"
        }
      ]
    }
  }
}
```

文本模块通常读取：

- `text`：文本内容。
- `path`：来源路径。
- `name`：来源名称。
- `sourceType`：来源类型，例如 `file` 或 `ocr`。

## 文件夹输入

文件夹输入会放在 `inputs.folder` 下。它不是集合，只代表一个文件夹。

```json
{
  "inputs": {
    "folder": {
      "dataType": "folder",
      "path": "D:/Project/Assets",
      "name": "Assets",
      "relativePath": "Assets"
    }
  }
}
```

文件夹模块通常读取：

- `path`：文件夹路径。
- `name`：文件夹名称。
- `relativePath`：相对待审核文件夹或产物文件夹的路径。根文件夹时为空字符串。

## files 字段

`files` 是当前步骤可处理的文件列表。

```json
{
  "files": [
    {
      "path": "D:/Project/assets/a.png",
      "name": "a.png",
      "extension": "png",
      "fileType": "image",
      "sourceAssetId": "asset_1",
      "sourceAssetName": "待测项目",
      "relativePath": "assets/a.png"
    }
  ]
}
```

如果流程给当前模块连接了图片或文本数据线，`files` 会尽量按数据线内容收窄。文件夹输入不会展开到 `files`，文件夹模块应该读取 `inputs.folder.path`。

## params 字段

`params` 来自 `module.json` 的 `parameters`，也就是用户在流程节点右侧填写的参数。

```json
{
  "params": {
    "threshold": 0.75,
    "riskWords": ["广告", "联系方式"],
    "strictMode": true
  }
}
```

模块不要自己猜默认值。默认值应该写在 `module.json` 里，客户端会把最终参数传进来。

## previous 字段

`previous` 保存上游步骤的原始输出。

```json
{
  "previous": {
    "ocr_step": {
      "status": "completed",
      "verdict": "pass",
      "outputs": {
        "fullText": "OCR 识别出的文字"
      }
    }
  }
}
```

如果只是消费图片或文本集合，优先使用 `inputs`。只有需要读取上游模块的完整原始结果时，再使用 `previous`。

## output.json 必填结构

模块结束前必须写出 `output.json`。推荐结构：

```json
{
  "status": "completed",
  "verdict": "review",
  "message": "命中 2 个风险文件。",
  "processedFiles": 12,
  "matchedFiles": 2,
  "artifactCount": 1,
  "reportSection": "### 文本合规检测\n\n命中 2 个风险文件，需要人工复审。"
}
```

字段说明：

| 字段 | 建议必填 | 说明 |
| --- | --- | --- |
| `status` | 是 | 当前步骤运行状态。 |
| `verdict` | 是 | 当前步骤审核结论。 |
| `message` | 是 | 给用户看的简短说明。 |
| `processedFiles` | 是 | 处理了多少个文件或条目。 |
| `matchedFiles` | 是 | 命中了多少个风险文件或条目。 |
| `artifactCount` | 否 | 生成了多少个产物文件。 |
| `reportSection` | 是 | 当前模块的报告片段。 |

客户端会保留 `output.json` 里的其他字段，后续步骤可以从 `previous` 里读取。

如果模块少写了部分字段，客户端会尽量用默认值兜底。但模块开发时不要依赖兜底，应该主动写全关键字段，方便报告展示、自动化验收和后续模块读取。

## 数据输出 outputs

如果模块在 `module.json` 里声明了 `dataOutputs`，需要在 `output.json` 的 `outputs.<handle>` 写出对应数据。图片集合示例：

```json
{
  "status": "completed",
  "verdict": "pass",
  "message": "已生成 12 张图片。",
  "processedFiles": 3,
  "matchedFiles": 0,
  "artifactCount": 12,
  "outputs": {
    "images": {
      "dataType": "imageCollection",
      "items": [
        {
          "path": "D:/UGCAuditArtifacts/task/run/关卡预览图/Level01/all_color.png",
          "name": "all_color.png",
          "extension": "png",
          "sourceAssetId": "ap-level-preview",
          "sourceAssetName": "AP-关卡预览图",
          "relativePath": "关卡预览图/Level01/all_color.png"
        }
      ]
    }
  },
  "reportSection": "### AP-关卡预览图\n\n已生成 12 张图片。"
}
```

下游模块通过数据线接收这个图片集合，不需要读取 `previous`。

## status 取值

推荐使用：

| `status` | 含义 |
| --- | --- |
| `completed` | 正常完成。 |
| `skipped` | 主动跳过。 |
| `cancelled` | 用户中断。 |
| `error` | 执行失败。 |

## verdict 取值

推荐使用：

| `verdict` | 含义 |
| --- | --- |
| `pass` | 通过。 |
| `review` | 需要人工复审。 |
| `reject` | 不通过。 |
| `error` | 执行失败。 |

如果模块没有写 `verdict`，客户端会按“需要人工复审”处理。建议始终显式写出。

## OCR 模块输出文本给后续模块

`image_ocr` 模块可以把识别结果传给后续 `text_safety` 模块。推荐在 `output.json` 里写 `results[].fullText`：

```json
{
  "status": "completed",
  "verdict": "pass",
  "message": "识别完成。",
  "processedFiles": 2,
  "matchedFiles": 2,
  "results": [
    {
      "path": "D:/Project/a.png",
      "name": "a.png",
      "relativePath": "a.png",
      "fullText": "第一张图片里的文字"
    },
    {
      "path": "D:/Project/b.png",
      "name": "b.png",
      "relativePath": "b.png",
      "fullText": "第二张图片里的文字"
    }
  ],
  "reportSection": "### OCR 识别\n\n已识别 2 张图片。"
}
```

客户端会把这些 `fullText` 转成文本集合，供后续文本审核模块使用。

如果只有一段总文本，也可以写：

```json
{
  "status": "completed",
  "verdict": "pass",
  "message": "识别完成。",
  "outputs": {
    "fullText": "识别出的全部文字"
  },
  "reportSection": "### OCR 识别\n\n识别完成。"
}
```

## 产物文件

模块如果生成截图、标注图、明细表等文件，建议写入 `stepArtifactDir`。

```python
artifact_dir = Path(input_data["stepArtifactDir"])
artifact_dir.mkdir(parents=True, exist_ok=True)
```

然后在 `output.json` 里更新：

```json
{
  "artifactCount": 1,
  "reportSection": "### 图片检测\n\n已生成标注图。"
}
```

如果要在报告里引用产物图片，`reportSection` 可以直接写图片路径。报告写法见 [客户端 Markdown 与 HTML 报告解析说明](./客户端Markdown与HTML报告解析说明.md)。

## 进度与中断

实时运行时，如果 `input.json` 里有 `progressPath`，模块可以追加 JSON 行：

```json
{"progress": 0.5, "message": "已处理 50 张图片", "processed": 50, "total": 100}
```

如果 `input.json` 里有 `cancelPath`，模块可以定期检查该文件是否存在。存在时尽快停止，并写出：

```json
{
  "status": "cancelled",
  "verdict": "review",
  "message": "任务已中断。",
  "processedFiles": 50,
  "matchedFiles": 0,
  "artifactCount": 0,
  "reportSection": "### 模块已中断\n\n用户中断了任务。"
}
```

耗时模块应该支持进度和取消，否则用户可能看到长时间没有反馈。完整写法见 [长任务进度与取消说明](./长任务进度与取消说明.md)。
