# 模块打包与交付说明

本文档说明模块开发完成后，应该交付哪些文件，以及交付前要检查什么。

## 交付目标

交付给其他开发者或使用者时，目标是让对方做到：

1. 拿到模块文件夹。
2. 在客户端“模块管理”里选择该文件夹导入。
3. 把模块拖到流程里。
4. 配置参数或模型路径。
5. 运行任务并得到报告。

## 推荐交付结构

```text
MyAuditModule/
  module.json
  main.py
  README.md
  icon.png
  Model/
    config.json
    model.bin
  assets/
    labels.json
  examples/
    input.demo.json
    output.demo.json
```

其中：

- `module.json` 必须存在。
- `main.py` 或其他入口文件必须存在。
- `README.md` 建议说明模块用途、参数含义、流程连接方式。
- `Model/` 只在模块需要本地模型时提供。
- `examples/` 可放调试输入输出样例。

## 不要交付的内容

不要把这些文件放进交付包：

- `__pycache__/`
- `.venv/`
- `.git/`
- 临时日志
- 本机运行历史
- 大型下载缓存
- 用户隐私数据
- 只在开发机有效的绝对路径配置

## 模型文件

如果模块需要模型，优先使用这个结构：

```text
MyAuditModule/
  module.json
  main.py
  Model/
    ...
```

并在 `module.json` 里提供 `modelPath` 参数：

```json
{
  "key": "modelPath",
  "name": "模型目录",
  "description": "选择本地模型所在目录。",
  "parameterType": "path",
  "defaultValue": "",
  "required": true,
  "options": []
}
```

如果模块目录下已经有 `Model/`，客户端会把它作为默认模型目录。模块代码仍然应该从 `input.json` 的 `modelPath` 字段读取，不要写死 `Model` 的绝对路径。

## 依赖文件

如果模块需要额外依赖，建议在 `README.md` 写清楚：

- 依赖名称。
- 版本要求。
- 安装方式。
- 是否需要 GPU。
- 首次运行大概需要多久。

如果依赖很重，不建议让模块运行时自动下载。更稳妥的做法是让交付包或部署说明提前准备好依赖。

依赖和运行环境的完整约定见 [模块依赖与运行环境说明](./模块依赖与运行环境说明.md)。

## 长任务要求

如果模块可能运行超过几秒，交付前要确认：

- 能写入进度，让客户端显示当前处理状态。
- 用户取消任务后，模块能尽快停止。
- 取消时也能写出合法 `output.json`。
- 报告里能说明任务处理到哪里。

进度和取消协议见 [长任务进度与取消说明](./长任务进度与取消说明.md)。

## 版本升级

升级模块时建议同步更新：

- `module.json` 里的 `summary`。
- `README.md` 的变更说明。
- 示例输入输出。
- 模型版本说明。

模块 ID 不建议频繁改变。只要它仍然是同一个模块，就保持相同 `id`。这样用户重新导入时，客户端可以用新文件夹记录替换旧记录。

## 导入和移除规则

客户端导入自定义模块时，记录的是模块源文件夹路径，不会复制整个模块目录。

这意味着：

- 导入后不要移动或删除模块源文件夹。
- 如果要换位置，重新导入新位置的模块文件夹。
- 移除导入模块通常只是取消注册，不会删除源文件夹。
- 如果模块已经在当前流程中使用，需要先从流程里删除相关步骤，再移除模块。

## 交付前检查清单

交付前逐项检查：

- `module.json` 是合法 JSON。
- `id`、`name`、`kind`、`icon` 已填写。
- `launch.command` 指向的文件存在。
- 图标文件存在，或使用了内置图标名。
- 参数默认值和参数类型匹配。
- 没有写死开发机绝对路径。
- 依赖和模型准备方式已写清楚。
- 本地运行入口脚本能生成 `output.json`。
- 示例输入输出放在 `examples/` 下。
- 在客户端能导入模块。
- 在流程中能连接正确的数据线。
- 实际运行后有 `output.json` 和 `result.md`。
- 报告页能显示 `reportSection`。
- 长任务能显示进度并响应取消。
- CLI 或客户端完整流程至少验收一次。
- 交付包里没有本机缓存、日志、隐私数据。

自动化验收方式见 [模块自动化验收与 CLI 说明](./模块自动化验收与CLI说明.md)。

## README 建议模板

```markdown
# 模块名称

## 用途

说明这个模块检查什么内容，适合接在哪类流程里。

## 流程连接

- 模块类型：text_safety
- 数据输入：文本集合
- 建议连接：待测项目中所有文本 --文本数据--> 本模块

## 参数

| 参数 | 说明 | 默认值 |
| --- | --- | --- |
| riskWords | 风险词列表 | 广告、联系方式 |

## 模型和依赖

说明是否需要模型，模型目录应该怎么放，是否需要额外依赖。

## 输出

说明 `verdict` 的判断规则，以及报告里会展示什么。
```
