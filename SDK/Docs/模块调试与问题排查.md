# 模块调试与问题排查

本文档说明模块怎么本地调试，以及常见问题怎么定位。

## 调试顺序

推荐按这个顺序排查：

1. 先确认 `module.json` 能被客户端导入。
2. 再用一个手写的 `input.json` 本地运行入口脚本。
3. 再在客户端里拖入流程，连好顺序线和数据线。
4. 运行任务后检查步骤目录里的文件。
5. 最后检查报告页展示是否符合预期。
6. 交付前用 CLI 或完整客户端流程再验收一次。

## 本地运行模块

可以先准备一个最小 `input.demo.json`：

```json
{
  "runId": "local_run",
  "stepId": "local_step",
  "moduleId": "custom.demo.text-risk",
  "moduleKind": "text_safety",
  "moduleName": "示例文本审核",
  "workDir": "./debug",
  "taskName": "本地调试",
  "artifactDir": "./debug",
  "stepArtifactDir": "./debug/artifacts",
  "resourceRoot": "./debug/resources",
  "inputs": {
    "texts": {
      "dataType": "textCollection",
      "items": [
        {
          "sourceType": "file",
          "path": "demo.txt",
          "name": "demo.txt",
          "relativePath": "demo.txt",
          "text": "这里包含广告内容"
        }
      ]
    }
  },
  "files": [],
  "params": {
    "riskWords": ["广告"]
  },
  "modelPath": "",
  "previous": {}
}
```

然后在模块目录运行：

```powershell
python main.py --input input.demo.json --output output.demo.json
```

检查 `output.demo.json`：

- 必须是合法 JSON。
- 必须包含 `status`、`verdict`、`message`、`reportSection`。
- `processedFiles` 和 `matchedFiles` 要符合预期。

## 客户端运行后看哪些文件

客户端每次运行模块时，会给每个步骤生成一个步骤目录。重点看：

| 文件 | 用途 |
| --- | --- |
| `input.json` | 客户端实际传给模块的数据。 |
| `output.json` | 模块实际写出的结果。 |
| `stdout.log` | 模块标准输出。 |
| `stderr.log` | 模块错误输出。 |
| `result.md` | 当前模块报告片段。 |
| `artifacts/` | 当前步骤生成的产物目录。 |

排查时优先看 `input.json` 和 `output.json`。如果模块没有启动成功，再看 `stderr.log`。

## 常见问题

| 问题 | 常见原因 | 处理 |
| --- | --- | --- |
| 导入失败，提示缺少 `module.json` | 选择的不是模块根目录。 | 选择包含 `module.json` 的文件夹。 |
| 导入失败，提示缺少 `id`、`name`、`kind` 或 `icon` | `module.json` 必填字段没写。 | 按 [模块文件夹与配置说明](./模块文件夹与配置说明.md) 补齐。 |
| 导入失败，提示图标文件不存在 | `icon` 写的是图片路径，但文件不存在。 | 修正图片路径，或改用内置图标名。 |
| 导入失败，提示不能导入内置模块 ID | 模块 ID 使用了系统保留 ID。 | 改成 `custom.公司名.模块名`。 |
| 模块启动失败，提示启动文件不存在 | `launch.command` 路径不对。 | 确认入口文件在模块目录下，或填写正确绝对路径。 |
| 模块运行后显示失败 | 没有写出 `output.json`，或 `output.json` 不是合法 JSON。 | 确保程序结束前写出 JSON 文件。 |
| 模块运行了但没有处理文件 | 没有连接数据线，或模块读取了错误输入口。 | 检查 `input.json` 里的 `inputs`。 |
| 长任务没有任何反馈 | 模块没有写 `progressPath`。 | 按 [长任务进度与取消说明](./长任务进度与取消说明.md) 写进度。 |
| 点击取消后还在运行 | 模块没有检查 `cancelPath`。 | 在循环中定期检查取消标记。 |
| 文本模块没有收到 OCR 文字 | OCR 输出没有 `fullText`。 | 在 OCR 输出中写 `results[].fullText` 或 `outputs.fullText`。 |
| 报告内容不显示 | `reportSection` 为空或 Markdown 写法有问题。 | 先看 `result.md`，再看报告文档。 |
| 模型路径为空 | 没有 `Model/` 目录，也没有在参数中配置 `modelPath`。 | 增加 `Model/` 或让用户填写模型目录。 |

## 如何判断问题在模块还是流程

看 `input.json`：

- `inputs` 里没有 `images` 或 `texts`：大概率是流程连接问题。
- `inputs` 里有数据，但模块输出处理数量为 0：大概率是模块读取输入的代码有问题。
- `params` 和界面填写不一致：检查 `module.json` 参数 key 是否写错。
- `modelPath` 为空：检查是否有 `modelPath` 参数，或模块目录下是否有 `Model/`。

看 `output.json`：

- 文件不存在：模块没有写结果，或启动失败。
- JSON 解析失败：输出格式错误。
- `status` 是 `error`：模块主动报告失败。
- `verdict` 不符合预期：模块自己的判断逻辑需要检查。

看 `stderr.log`：

- `ModuleNotFoundError`：Python 依赖缺失。
- `FileNotFoundError`：读取了不存在的路径。
- `PermissionError`：没有权限读写目标路径。
- `Traceback`：Python 代码异常。

## 报告排查

模块报告最终来自 `reportSection`。建议先在 `output.json` 里确认它存在，再看 `result.md`。

报告写法见 [客户端 Markdown 与 HTML 报告解析说明](./客户端Markdown与HTML报告解析说明.md)。

建议：

- 模块报告从三级标题开始，例如 `### 图片合规检测`。
- 不要生成完整报告，只生成当前模块自己的片段。
- 图片、表格、流程图先用小样例测试。

## CLI 验收排查

如果本地试跑通过，但完整流程不对，可以用 `ugc-audit.exe run` 验收保存好的方案：

```powershell
ugc-audit.exe run --scheme "D:\AuditSchemes\demo.ugcaudit" --input "D:\UGCProject" --output "D:\AuditRuns\demo"
```

然后检查输出目录里的：

- `cli-result.json`：看退出码、状态和结论。
- `run.json`：看每个步骤的真实结果。
- `report.md`：看最终报告。

完整说明见 [模块自动化验收与 CLI 说明](./模块自动化验收与CLI说明.md)。

## 退出码说明

客户端最终以 `output.json` 为准。模块即使异常退出，只要已经写出了合法 `output.json`，客户端仍可以读取其中的结果。

建议：

- 正常完成时写 `status: "completed"`。
- 可预期失败时也尽量写出 `output.json`，并设置 `status: "error"` 和清晰的 `message`。
- 不要只依赖进程退出码表达业务结论。
